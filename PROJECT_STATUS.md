## 2. Главная незакрытая проблема

Railway production container нестабильно/неуспешно подключается к:

`smtp.mail.ru:587`

Ошибки:

- `ESOCKET`
- `ETIMEDOUT`
- `command=CONN`

Ошибка возникает до SMTP greeting, авторизации и принятия получателя.

Mail.ru письмо не принимает, потому что TCP-соединение из Railway не устанавливается.

Успешная ручная отправка выполнялась локально с Mac, а не из Railway.

Следовательно:

- SMTP credentials правильные.
- Пароль приложения правильный.
- Шаблон письма работает.
- Проблема не в e-mail покупателя.
- Проблема не в FazerCards.
- Проблема не в Alfa.
- Проблема в сетевом пути Railway -> `smtp.mail.ru`.

## 4. Принятое решение

Основной и единственный способ выдачи товара:

E-mail покупателя.

Не делать сейчас:

- выдачу через MAX ID;
- Telegram;
- личный кабинет;
- возврат кода через frontend;
- домен;
- SMS.

## 5. Задача на следующий рабочий день

Главная задача:

Убрать прямую SMTP-отправку Railway -> Mail.ru.

Реализовать надёжный почтовый шлюз по HTTPS.

Предпочтительный вариант:

Railway backend
-> защищённый HTTPS-запрос
-> существующий VPS пользователя
-> VPS отправляет письмо через `smtp.mail.ru`
-> покупатель получает письмо.

Требования к шлюзу:

- отдельный HTTPS endpoint;
- секретный токен между Railway и VPS;
- принимать только:
  - order number,
  - e-mail получателя,
  - название товара,
  - номинал,
  - цифровой код,
  - текст/HTML письма;
- не логировать цифровой код;
- SMTP credentials хранить только на VPS;
- timeout и retry;
- idempotency по `order.id`;
- повторная отправка письма не должна повторно покупать товар;
- сохранять результат отправки в PostgreSQL:
  - `email_status`,
  - `email_sent_at`,
  - `email_message_id`,
  - `email_error`.

## 6. План запуска

1. Проверить доступ VPS к `smtp.mail.ru:587`.
2. Отправить тестовое письмо с VPS.
3. Создать защищённый HTTPS mail gateway.
4. Подключить Railway к gateway.
5. Повторно отправить сохранённые коды неудачных заказов.
6. Убедиться, что FazerCards повторно не вызывается.
7. Провести один новый реальный заказ за минимальную сумму.
8. Подтвердить полный автоматический цикл:

оплата
-> Alfa `OrderStatus=2`
-> FazerCards
-> код сохранён
-> HTTPS gateway
-> письмо получено покупателем.

# Экран успешной оплаты (июль 2026)

## Что реализовано

- Создан отдельный экран состояния заказа после оплаты.
- Экран поддерживает несколько состояний:
  - проверка оплаты;
  - заказ обрабатывается;
  - письмо отправлено;
  - письмо отправляется;
  - ошибка отправки письма;
  - отменённый/неоплаченный заказ.
- Frontend больше не получает цифровой код.
- Frontend больше не вызывает fulfillment.
- Используется безопасный `GET /api/orders/:id`.
- `customerEmail` заменён на `customerEmailMasked`.
- `buildOrderReturnUrl` переведён на безопасное формирование через `URL` API.

## Что проверено

- Frontend успешно собирается.
- Backend проходит `node --check`.
- Реальный production-заказ успешно выполнен.
- Alfa работает.
- FazerCards работает.
- Письмо Mail.ru приходит.
- Цифровой код доставляется корректно.

## Что обнаружено

Во время production-проверки установлено:

`ALFA_RETURN_URL` сейчас указывает на

`/api/payment/success`

Этот backend route делает redirect на

`/?payment=success`

При этом параметр `mdmOrderId` теряется.

Из-за этого frontend открывает каталог вместо нового экрана статуса заказа.

Сам экран реализован корректно, но пользователь до него не доходит.

## Подтверждённая причина

Причина находится в legacy redirect backend route `/api/payment/success`.

Экран статуса не является причиной проблемы.

## Текущее решение

Пока экран оставлен в проекте.

Он не удаляется.

Архитектура сохранена.

Возврат к доработке будет выполнен позднее после выбора окончательной стратегии.
